# 開発環境を「コードで」管理する

これまでのコマンド（`nix profile install` など）はMac全体に対する操作でした。

ここから扱うのは「**プロジェクトごとに独立した開発環境**」です。

```text
~/projects/A/    → Node 18 + Python 3.11
~/projects/B/    → JDK 21 + Maven
~/projects/C/    → Rust + Postgres
```

それぞれが完全に分離していて、ディレクトリに入ると自動的にツールが切り替わる ——
この体験が Nix のキモです。

---

# 一時的に試す — nix shell -p

「Node.jsをちょっと試したい」程度なら `nix shell -p`:

```bash
$ nix shell nixpkgs#nodejs_20 nixpkgs#yarn
$ node --version
v20.x.x
$ exit
```

ただ毎回コマンドを書くのは面倒。**設定ファイルに書いておく** のが本筋です。

---

# shell.nix — 旧式の書き方

ファイル名 `shell.nix` を置くと、`nix-shell` で読み込めます。

```nix
{ pkgs ? import <nixpkgs> {} }:

pkgs.mkShell {
  buildInputs = with pkgs; [
    nodejs_20
    yarn
    jq
  ];
}
```

`buildInputs` に欲しいパッケージを並べるだけ。

```bash
$ nix-shell        # ← カレントディレクトリの shell.nix を読み込む
[nix-shell]$ node --version
v20.x.x
```

> [note] `<nixpkgs>` は古い書き方で、グローバルなチャンネルを参照しています。次章以降の Flakes ではここを明示的に固定します。

---

# モダンな書き方 — nix develop と flake.nix

Flakes 時代の標準は `flake.nix` + `nix develop` です。

```bash
$ nix develop      # ← カレントディレクトリの flake.nix を読み込む
```

シンタックスは shell.nix とは少し違いますが、本質は同じ。Flakesは第6章で詳しくやるので、今は **「shell.nix の進化版がある」** くらいの理解でOK。

---

# devShell の基本形（先取り）

flake.nix での devShell はこんな感じです:

```nix
{
  description = "My dev environment";
  inputs.nixpkgs.url = "github:NixOS/nixpkgs/nixos-unstable";

  outputs = { self, nixpkgs }:
    let
      system = "aarch64-darwin";  # Apple Silicon Mac
      pkgs = import nixpkgs { inherit system; };
    in {
      devShells.${system}.default = pkgs.mkShell {
        buildInputs = with pkgs; [ nodejs_20 yarn jq ];
      };
    };
}
```

詳細は気にしなくてOK。**第6章で構造を分解します**。

---

# shellHook で起動メッセージ

`mkShell` には `shellHook` というフィールドがあって、env入った時に実行されます。

```nix
pkgs.mkShell {
  buildInputs = with pkgs; [ nodejs_20 ];
  shellHook = ''
    echo "🚀 Node.js dev environment loaded"
    echo "node $(node --version), npm $(npm --version)"
  '';
}
```

env変数の設定にも便利です:

```nix
shellHook = ''
  export DATABASE_URL=postgresql://localhost/dev
'';
```

---

# direnv 連携 — 自動切り替えの最終形

毎回 `nix develop` を叩くのは面倒。**direnv** を使えば、ディレクトリに `cd` するだけで自動切り替えできます。

```bash
# direnv をインストール（一度だけ）
$ nix profile install nixpkgs#direnv nixpkgs#nix-direnv

# プロジェクトに .envrc を置く
$ echo "use flake" > .envrc
$ direnv allow
```

これで `cd ~/projects/A/` した瞬間に node が使える状態になります。

> [hint] direnv はNixのキラー連携。これを覚えると「環境を毎回手で立ち上げる」概念が消えます。

---

# nix-shell vs nix develop の比較

| 観点 | nix-shell (shell.nix) | nix develop (flake.nix) |
|------|----------------------|-------------------------|
| バージョン固定 | チャンネル依存 | flake.lock で完全固定 |
| シンタックス | シンプル | やや冗長だが構造化 |
| 学習コスト | 低 | 中 |
| 公式の方向 | レガシー | **モダン推奨** |

新しく作るなら **flake.nix を選びましょう**。

# 演習 — shell.nix を書いてみる

> [try] 任意のディレクトリで以下の `shell.nix` を作り、`nix-shell` で入ってみてください。
>
> ```nix
> { pkgs ? import <nixpkgs> {} }:
> pkgs.mkShell {
>   buildInputs = with pkgs; [
>     python3
>     nodejs_20
>     jq
>   ];
>   shellHook = ''
>     echo "Hello from nix-shell!"
>   '';
> }
> ```
>
> ```bash
> $ nix-shell
> Hello from nix-shell!
> [nix-shell]$ python3 --version
> [nix-shell]$ node --version
> [nix-shell]$ jq --version
> ```
>
> 抜けるときは `exit` で出られます。

次の章では Nix言語そのものを学んで、`shell.nix` の中身を読めるようにします。
